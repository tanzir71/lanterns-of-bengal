About the Game

Lanterns of Bengal is a slow, luminous walk through village nights—lanterns glowing like punctuation in the dark, whispered folklore at the edge of a tea stall, boat rides that feel endless, and the ambient chorus of insects and water.

It is less about winning and more about remembering: the way light moves, the way the river listens back, the way fear and comfort sit in the same frame.

Inspiration & Vision

This project brings lived experience into a format that can travel. Village nights, river life, and folklore are not just aesthetics here—they are the emotional map.

I wanted Dhaka-born youth to recognize the texture of the night, and Western players to step into it without a museum voice. The goal is intimacy, not translation.

How This Was Built (Tools & AI)

I approached this like a designer first: mood, structure, intention. I did not start with code—I started with the feeling of a lantern moving through a landscape that listens back.

Sprites were generated with Gemini, then “vibe-coded” into the game using Codex, iterating until the tone felt right.

Why This Is an MVP

Current AI-assisted game building and sprite generation are still in their infancy. Auto-generated assets often need heavy human curation and iteration—cleanup, style alignment, pacing fixes, and the subtle work that makes a world feel authored.

This project is intentionally an early prototype, documenting the process more than final polish.

What’s Next

  • Explore an isometric, open-world direction that can hold a fuller range of lived experience in South Asia—a “GTA meets The Sims” mood, grounded in daily life rather than spectacle.
  • Prototype deeper vehicles: operating boats, rickshaws, buses, and motorcycles with tactile controls.
  • Expand into a life-sim layer: routines, relationships, small economies, and local spaces that feel real.

Realistically, that larger vision will require significant AI advances to become achievable at indie scale—more reliable asset generation, better world-building tooling, and stronger consistency over time.

Code & Technical Notes

This prototype is intentionally straightforward: one canvas game loop, a lightweight in-browser editor, and local persistence. The goal is something you can read, remix, and evolve.

Project structure
  • index.html — landing page + canvas container + editor UI.
  • style.css — page + editor styling (dark UI, minimal chrome).
  • game.js — game loop, rendering, input, editor, audio, persistence.
  • assets/ — sprite sheets and audio files used by the prototype.
The core loop (how the game runs)

The game uses requestAnimationFrame. Each frame runs: update(dt) → updateAudioState() → draw(t). 

function loop(now) {
  const dt = Math.min(0.033, (now - lastTime) / 1000 || 0.016);
  update(dt, now / 1000);
  updateAudioState();
  draw(now / 1000);
  requestAnimationFrame(loop);
}

The update step handles movement, boats, collisions, ghosts, collectibles, and camera. The draw step layers background, world, entities, and HUD in a consistent order.

Levels, persistence, and the built-in editor
  • Levels are editable and can be exported/imported as JSON. Progression cycles through named levels (e.g., “Level 1”, “Level 2”).
  • Levels and editor state are saved locally (localStorage), so your tweaks persist between refreshes (river-lantern-levels-v2, river-lantern-level-v1).
  • Custom sprites are stored in the browser (IndexedDB), so you can replace props without a build step.
  • Audio replacements are also stored locally (IndexedDB), so a “soundscape pass” can happen inside the prototype without rebuilding.
  • The editor can adjust assets, platforms, props, water bounds, and audio settings, then save them locally.

This is designed for rapid iteration: you can treat the game like a living mood-board, then harden it later.

How to repurpose or improve this prototype
  • Re-skin the world by swapping the sprite sheets referenced in imagePaths inside game.js.
  • Turn it into a reusable engine by splitting game.js into modules (input, renderer, physics, audio, editor, persistence) and adding a small data schema for levels.
  • Improve “feel” by tuning just a few constants: gravity, run speed, jump strength, boat steering, and ghost spawn cadence.
  • Add scalability by introducing a fixed-timestep update, texture atlases, and a lightweight build pipeline once iteration stabilizes.

Create New Asset

Upload Guide

  • Use PNG or WebP when possible, with a transparent background around the sprite.
  • Upload one complete object per file: one tree, one platform block, one collectible, one enemy, or one boat.
  • Avoid screenshots of full sprite sheets, UI labels, guide text, watermarks, shadows baked into a dark rectangle, or cropped-off edges.
  • Leave a small transparent margin around the silhouette so glow and antialiasing are not cut off.
  • For platforms, make the top edge clear and mostly flat; the game lets the hero stand on the rectangle you place.
  • For collectibles, keep the object centered and readable at small size, roughly 32 to 48 pixels tall after scaling.
  • For enemies and traversal assets, upload the whole side-view figure or boat in a single frame. This editor creates static assets; animation sheets need manual crop setup.
